4. Token issuance (login / register)

sequenceDiagram
  participant C as Client
  participant API as Auth API
  participant AS as AuthService
  participant M as MongoDB
  participant R as Redis

  C->>API: POST login or register
  API->>AS: login or register
  AS->>M: find or create user
  AS->>M: read tokenVersion
  AS->>AS: sign access JWT new jti
  AS->>AS: sign refresh JWT new familyId
  AS->>M: INSERT refresh_sessions
  AS->>R: SET rt jti with TTL
  AS-->>API: user and tokens
  API-->>C: return access and refresh tokens

Multi-device: each login adds a new refresh_sessions row + new rt:{jti}; old devices unchanged.

5. Protected API — access token validation

flowchart TD
  A["Bearer access JWT"] --> B{"Token present?"}
  B -->|No| E1["401 Missing bearer token"]
  B -->|Yes| C["verifyAccessToken HS256 and exp"]
  C -->|Fail| E2["401 Invalid access token"]
  C --> D{"typ is access?"}
  D -->|No| E2
  D --> F["Redis GET abl jti"]
  F -->|exists| E3["401 Token revoked"]
  F --> G["Mongo findById sub"]
  G -->|missing| E4["401 User not available"]
  G --> H{"isActive?"}
  H -->|No| E4
  H --> I{"tokenVersion matches tv?"}
  I -->|No| E5["401 Token version mismatch"]
  I --> J["Set req.auth from DB"]
  J --> K{"authorize middleware?"}
  K -->|role mismatch| E6["403 Insufficient role"]
  K --> L["Controller business logic"]

Note: roles on req.auth come from Mongo, not only from JWT.

6. Refresh token rotation

sequenceDiagram
  participant C as Client
  participant API as Auth API
  participant AS as AuthService
  participant R as Redis
  participant M as MongoDB

  C->>API: POST refresh
  API->>AS: refresh
  AS->>AS: verifyRefreshToken
  AS->>R: check family revoked
  alt family revoked
    API-->>C: 401 family revoked
  end
  AS->>R: GETDEL rt jti
  alt Redis miss
    AS->>M: find session by jti
    AS->>M: revoke row
  end
  alt session missing or reuse
    AS->>R: mark family revoked
    API-->>C: 401 reuse detected
  end
  AS->>M: load user check tv
  AS->>AS: issueAuthTokens same familyId
  AS->>M: new refresh_sessions row
  AS->>R: SET new rt jti
  API-->>C: new access and refresh tokens

Old refresh JWT is single-use; replay revokes entire family.

7. Logout flows

7a. Single device — POST /logout

flowchart LR
  subgraph access ["Access token"]
    A1["authenticate passes"] --> A2["Redis SET abl jti TTL"]
  end
  subgraph refresh ["Refresh token"]
    R1["verify refresh JWT"] --> R2["Mongo revoke session"]
    R2 --> R3["Redis DEL rt jti"]
  end
  TV["tokenVersion unchanged"]

Other devices keep working.

7b. All devices — POST /logout-all

flowchart TD
  A["authenticate"] --> B["Optional blacklist access abl jti"]
  B --> C["Mongo revoke all refresh_sessions"]
  C --> D["Redis DEL all rt jti for user"]
  D --> E["Mongo tokenVersion increment"]
  E --> F["All access JWTs with old tv fail"]
  E --> G["All refresh JWTs with old tv fail"]
StoreRefresh on logout-allAccess on logout-all
MongoRows revoked, not deletedtokenVersion++
RedisAll rt:{jti} deletedOnly current abl:{jti} (others die via tv)

Same pattern for change-password and reset-password.

8. Password reset flow

sequenceDiagram
  participant C as Client
  participant API as Auth API
  participant AS as AuthService
  participant M as MongoDB

  C->>API: POST forgot-password
  API->>AS: forgotPassword
  AS->>M: find user
  alt user exists
    AS->>M: INSERT password_reset_token
    Note over AS: dev only return resetToken
  end
  API-->>C: generic success message

  C->>API: POST reset-password
  API->>AS: resetPassword
  AS->>M: validate token mark used
  AS->>M: update password
  AS->>M: revoke all refresh_sessions
  AS->>AS: revokeAll Redis rt keys
  AS->>M: increment tokenVersion
  API-->>C: success

Production still needs an email provider (not in repo).

PropertyHow
Password safetybcrypt on User model

Instant global logouttokenVersion + revoke all sessions
No email enumerationSame forgot-password response
